# PopLua

PopLua is a Lua runtime for C# focused on source-generated bindings,
AOT-friendly execution, and sandboxed scripts.

## Minimal Usage

Install:

```bash
dotnet add package PopLua --version 1.0.0-rc.1
```

```csharp
using PopLua.Binding;
using PopLua.Context;
using PopLua.Exceptions;
using PopLua.Marshaling;
using PopLua.Runtime;
using PopLua.Sandboxing;

var lua = Engine.Create();

await using var session = lua.Session(Sandbox.Untrusted);
var result = await session.Run<long>("return 21 * 2");

Console.WriteLine(result.Unwrap());
```

## Generated Bindings

Projects that define PopLua generated bindings, such as `[Module]`,
`[Fn]`, or `[Userdata]`, must enable unsafe blocks because the source
generator emits Lua C callback function pointers:

```xml
<AllowUnsafeBlocks>true</AllowUnsafeBlocks>
```

Projects that only consume `Engine` to run scripts and do not declare
generated bindings do not need this setting.

Production applications can keep unsafe code out of the main app project by
placing generated bindings in a separate class library, for example
`MyApp.Scripting`, and enabling unsafe blocks only in that binding project.

NativeAOT publish has been validated for linux-x64 runtime-only consumers,
generated module bindings, generated userdata bindings, and async generated
bindings. Published apps still need a compatible Lua 5.4 or 5.5 library
available at run time. Windows and macOS are not part of the RC validation
claim.

Generated modules and userdata are registered explicitly:

```csharp
var lua = Engine.Create(b => b.Module<MathModule>());

[Module("mathx")]
public partial class MathModule
{
    [Fn("add")]
    public static long Add(long left, long right) => left + right;
}
```

Userdata bindings use `[Userdata]` and are returned by generated module
functions:

```csharp
[Module("vec")]
public partial class VecModule
{
    [Fn("new")]
    public static Vec2 New(double x, double y) => new(x, y);
}

[Userdata("vec2")]
public partial class Vec2(double x, double y)
{
    [Prop("x", ReadOnly = true)]
    public double X { get; } = x;

    [Fn("length")]
    public double Length() => Math.Sqrt(X * X + Y * Y);
}
```

Lua scripts normally call module functions with `.` and userdata instance
methods with `:`:

```lua
local v = vec.new(3, 4)
return v:length()
```

PopLua consumes the Lua userdata receiver internally and exposes it to C# as
the instance (`this`). Do not declare a `Value self` parameter on C#
userdata methods; use `Value` only for ordinary Lua values where supported.

Async module functions and userdata instance methods are supported with
`ValueTask` and `ValueTask<T>`:

```csharp
[Fn("fetch", Async = true)]
public async ValueTask<string> Fetch([Context] ScriptContext ctx, string id)
{
    await Task.Delay(10, ctx.Cancellation);
    return id;
}
```

Async operation faults are surfaced as Lua errors by the coroutine wrapper, so
Lua `pcall` can catch them when the sandbox exposes `pcall`. Host cancellation
remains terminal for the active execution. Generated async bindings use
`ValueTask` / `ValueTask<T>`; `Task` / `Task<T>` are intentionally unsupported.

## API Tooling Outputs

Generated binding projects produce API tooling files by default. The build
writes `poplua.api.json`, `poplua.d.lua`, and `poplua-api.md` to
`$(PopLuaApiOutputDir)`, which defaults under `obj/.../poplua-api`. Set
`PopLuaApiOutputDir` for CI artifacts or editor integration.
Set `PopLuaGenerateApiManifest`, `PopLuaGenerateLuaLsDefinitions`, or
`PopLuaGenerateApiDocs` to `false` to opt out of a specific output.

Manual generation is also possible from generated providers:

```csharp
File.WriteAllText("poplua.api.json", PopLua.Generated.PopLuaApiManifestProvider.Json);
File.WriteAllText("poplua.d.lua", PopLua.Generated.PopLuaLuaLsDefinitionProvider.Lua);
File.WriteAllText("poplua-api.md", PopLua.Generated.PopLuaApiDocumentationProvider.Markdown);
```

The provider classes are generated by default unless the matching generation
property is set to `false`. Providers are internal to the generated-binding
assembly, so manual export code must run in that same assembly. XML summaries,
remarks, parameter docs, return docs, examples, and exception docs flow into the
manifest and Markdown output when available.

For VS Code LuaLS, point your workspace at the generated `poplua.d.lua`
definition file, for example through `Lua.workspace.library`.

## Native Lua Requirement

PopLua uses an internal P/Invoke bridge to Lua 5.4 or 5.5. The host environment must
provide a compatible native library available under a supported platform
library name. PopLua prefers 5.5 when both are installed; set
`POPLUA_LUA_VERSION=5.4` or `5.5` to require a version.
On Arch Linux, install Lua 5.5 with `pacman -S lua`. Ubuntu 24.04 uses
`apt-get install liblua5.4-dev`. Native binaries are not bundled.

## Sandbox Basics

`Sandbox.Untrusted` starts with no capabilities and does not open standard Lua
libraries. Use it, or a stricter custom sandbox, for user-authored scripts.
Custom sandboxes can opt into selected native Lua standard libraries:

```csharp
var sandbox = Sandbox.Build(b => b.AllowSafeLibs());
```

The safe profile exposes selected base helpers plus native `math`, `string`,
`table`, and `utf8`. It does not open `io`, `os`, `package`, `debug`, or Lua
filesystem loading. `Sandbox.Trusted` opens all standard libraries and is
intended for trusted host-owned scripts only. Modules can declare a required
capability and are skipped when the active sandbox does not allow it. Capability
names are host-defined labels; they do not grant file, network, or process
access unless your host APIs enforce them.

```csharp
var sandbox = Sandbox.Build(b => b.Allow(Caps.FileRead));
await using var session = lua.Session(sandbox);
```

Instruction, active-time, wall-time, memory, GC threshold, and Lua call-depth
quotas are available through `Builder`:

```csharp
var sandbox = Sandbox.Build(b => b.Quota(
    instructions: 100_000,
    activeTime: TimeSpan.FromSeconds(1),
    wallTime: TimeSpan.FromSeconds(30)));
```

`activeTime` counts active Lua execution and host binding work. Generated async
functions pause active-time while actually suspended by default; set
`[Fn(Async = true, PauseTime = false)]` for async host work whose suspended
time should keep consuming active-time. `wallTime` always counts total elapsed
execution lifetime, including waits.

## Compile/Cache Workflow

For submitted scripts, create a named `Chunk`, compile it to `Bytecode`,
store the bytecode in your host application, and execute it later in a fresh
`Session` with the trigger identity, services, and sandbox.

Declare stable script-visible APIs as generated modules and userdata. For
context-like roots such as `ctx`, expose a `[Module("ctx")]` and read
per-execution host state from `[Context]` and `Services`. Use computed
module properties for values such as `ctx.author`, and module functions for
actions such as `ctx.reply(...)`. PopLua does not support constructing public
Lua API surfaces through dynamic global mutation.

Generated functions can accept structured Lua table descriptors as C# DTO
parameters. Descriptor shapes are rooted in exposed signatures and flow into the
manifest, LuaLS definitions, and Markdown docs. C# `required` descriptor
properties must be present in Lua tables, and nested descriptor errors include
field paths.

Output-only DTOs can opt into generated copy-based returns with `[Table]`.
Public readable properties, nested `[Table]` DTOs, and bounded lists are copied
into fresh Lua tables; raw Lua table handles are not exposed.

For reusable Lua modules, opt into controlled loading with
`EngineBuilder.Require`. The resolver receives the current `ScriptContext` and
a normalized dot-separated module name, then returns a host-approved `Chunk`
or `null`. PopLua does not use `package.path`, `package.cpath`, or filesystem
loading by default.

## Script Output

For user-authored scripts, prefer an explicit host module such as `log.info`,
`log.warn`, and `log.error` instead of relying on global `print`. This works in
`Sandbox.Untrusted`, can be capability-gated, and lets the host attach identity
and route output to its own logging or storage system.

Host-managed `require` is synchronous, opt-in, and resolver-backed. Async
resolvers, package managers, mod manifests, and automatic filesystem search are
not included in this preview.

## Production Checklist

- Use `Sandbox.Untrusted`, or a stricter custom sandbox, for user-authored scripts.
- Name submitted scripts and compile source to `Bytecode`.
- Do not accept arbitrary external Lua bytecode.
- Create fresh sessions per isolated trigger unless shared Lua state is intentional.
- Use `FunctionRef` only for callbacks that stay within the owning live
  session. For later external interactions, store host-side callback IDs or
  routing data instead of Lua closures.
- Inject identity, services, sandbox, and cancellation per execution.
- Collect diagnostics and metrics.
- Treat `Caps.*` as labels; host APIs must enforce real IO/network/process policy.
- Provide host-controlled logging/output through generated modules.
- Dispose sessions and provide a compatible Lua 5.4 or 5.5 native library.

## Errors And Diagnostics

Script failures are returned through `Result.Error`. Runtime
`ScriptException` values include chunk, line, and Lua traceback information
when Lua provides it. Configure `IDiagnostics` on `EngineBuilder` to
observe started/completed/failed executions, quota blocks, sandbox blocks, and
final execution metrics.
