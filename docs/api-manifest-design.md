# API Manifest Design

This note defines PopLua's API Manifest architecture for autocomplete,
generated documentation, API discovery, and future compatibility analysis.
The JSON manifest, LuaLS/LuaCATS definitions, and Markdown documentation
outputs are generated by default for generated-binding projects. Sandbox-profile
projection remains future work; compatibility comparison belongs to external
manifest tooling.

The manifest must stay aligned with PopLua's existing architecture:

- AOT-first.
- Source-generator driven.
- No runtime reflection or runtime API scanning.
- No public Lua interop surface.
- One package for normal users.
- One canonical metadata model, not a second binding system.

## Current Generator Inputs

The current generator already walks Roslyn syntax trees, resolves declared type
symbols, reads PopLua attributes, validates supported marshaling types, and
emits bindings. That means most manifest data can be collected from the same
symbols that already drive binding generation.

Available today:

| Data | Availability | Current source |
|---|---|---|
| Module name | Available | `[Module("name")]`. |
| Module capability | Available | `[Module(Cap = "...")]`. |
| Userdata name | Available | `[Userdata("name")]`. |
| Userdata options | Available | `Setters`, `ToString`, `Gc`. |
| Functions | Available | `[Fn]` methods. |
| Function Lua names | Available | explicit names or generated snake_case. |
| Async flag | Available | `[Fn(Async = true)]`; module-function runtime bridge implemented. |
| Parameters | Available | `IMethodSymbol.Parameters`. |
| Parameter names | Available | `IParameterSymbol.Name`. |
| Parameter types | Available | `IParameterSymbol.Type`. |
| Return types | Available | `IMethodSymbol.ReturnType`. |
| Context injection | Available | `[Context]`. |
| Variadic parameters | Available | final `Value[]`. |
| Constants | Available | `[Const]` fields/properties on modules. |
| Properties/fields | Available | `[Prop]` fields/properties and computed module property methods. |
| Readonly flag | Available | `[Prop(ReadOnly = true)]`. |
| Writable userdata members | Available | public setters and writable fields. |
| Operators | Available | `MethodKind.UserDefinedOperator`. |
| Operator metamethods | Available | generator maps C# operators to Lua names. |
| Ignore markers | Available | `[Ignore]`. |
| Namespace/type identity | Available | `INamedTypeSymbol` and display formats. |
| Diagnostics metadata | Available | generator-owned `DiagnosticDescriptor`s. |

Available in the internal manifest foundation:

- Schema/version constants.
- Stable Lua-facing API id helpers for modules, userdata, and members.
- Canonical `LuaApiType` and primitive/userdata type mapping.

Available in generated provider outputs:

- XML documentation comments via `ISymbol.GetDocumentationCommentXml(...)`.
- Summary, remarks, parameter descriptions, return descriptions, examples, and
  exception descriptions after parsing XML doc comment XML.
- Source symbol metadata for diagnostic/tooling references.

Limitations:

- Documentation for external referenced symbols is only available when Roslyn has
  a documentation provider for that reference; PopLua should treat external XML
  docs as best effort.
- Default parameter values are currently not part of binding semantics and should
  not be represented as callable optional Lua parameters yet.
- Type-parameter documentation is not represented yet because generated Lua
  bindings do not expose generic Lua-facing APIs in this preview.
- Obsolete/deprecation metadata is not represented yet.
- Runtime values and service instances are not visible to the generator.
- Standard Lua libraries are not generated from user code and need a separate
  PopLua-owned static catalog.
- Lua `pcall` does not reliably catch synchronous generated managed callback
  failures. This is a stable `1.0` runtime caveat rather than manifest metadata:
  PopLua fails the `Result` after the managed callback returns, avoiding
  native longjmp across managed frames. Async task faults are raised by the Lua
  coroutine wrapper and are catchable by `pcall`.

## Canonical Model

PopLua should introduce one internal generator model that becomes the canonical
representation of the exposed Lua API:

```text
LuaApiManifest
  ManifestVersion
  PopLuaVersion
  AssemblyIdentity
  GeneratedAt?           // optional, omitted from compatibility comparisons
  Modules[]
  Userdata[]
  Libs[]
  Diagnostics[]
```

Generated bindings, JSON manifests, LuaLS/LuaCATS definitions, Markdown docs, and
future compatibility tooling should all originate from this model.

The generator currently builds separate small records such as `ModuleInfo`,
`MethodInfo`, `ValueMemberInfo`, and `OperatorInfo`. Those should be replaced or
wrapped by a richer semantic model before code emission:

```text
LuaApiId
  Value                 // stable logical id, e.g. module:vec.new

LuaApiType
  Kind                  // nil, boolean, integer, number, string, userdata,
                        // lua-value, variadic, multiple-return, void,
                        // future-extension
  LuaName               // display name for Lua/docs
  DotnetName?           // source type identity for diagnostics/diff context
  UserdataId?           // stable id when Kind == userdata
  ElementType?          // arrays/future sequences
  ReturnTypes[]         // multiple-return values
  IsNullable
  IsLossy               // true when represented as any/unknown for an emitter

LuaApiSourceSymbol
  DotnetDisplayName      // e.g. MyApp.Vec2.Length()
  MetadataName           // stable CLR-ish symbol name where available
  ContainingType
  Namespace
  AssemblyName
  Location?              // optional and debug-oriented

LuaApiParameter
  Id
  Name
  Type
  IsContext
  IsVariadic
  Documentation?

LuaApiReturn
  Id
  Type
  Documentation?

LuaApiFunction
  Id
  LuaName
  CsName
  SourceSymbol
  Kind                  // module-function, userdata-method
  IsStatic
  IsAsync
  Parameters[]
  Returns[]
  Capability?
  Documentation
  Source

LuaApiValue
  Id
  LuaName
  CsName
  SourceSymbol
  Kind                  // const, property, field
  Type
  IsReadOnly
  IsWritable
  Documentation
  Source

LuaApiOperator
  Id
  Metamethod            // __add, __sub, __mul, __div, __unm, __eq, __lt, __le
  CsName
  SourceSymbol
  Parameters[]
  Return
  Documentation
  Source

LuaApiModule
  Id
  LuaName
  CsName
  SourceSymbol
  Namespace
  Capability?
  Functions[]
  Values[]
  Documentation
  Source

LuaApiUserdata
  Id
  LuaName
  CsName
  SourceSymbol
  Namespace
  Setters
  ToString
  Gc
  Methods[]
  Properties[]
  Operators[]
  Documentation
  Source

LuaApiDiagnostic
  Id
  Title
  MessageFormat
  Severity
  Category
```

`SourceSymbol` should explain where an API came from, but it is not the API's
stable identity. Diff and documentation tools should use `Id` as the Lua API
identity and `SourceSymbol` only for source context. This lets C# refactors keep
compatibility when the Lua-facing API is unchanged.

Benefits:

- Binding generation and documentation cannot drift semantically.
- Future outputs reuse validation, type mapping, name conversion, and diagnostics.
- Compatibility analysis can diff stable metadata instead of generated C# text.
- Sandbox-aware output can filter the same model by capabilities.

Risks:

- The generator becomes more complex.
- The model must be carefully separated into semantic collection, validation, and
  rendering phases.
- Public manifest APIs, if added too early, could freeze metadata shape before
  async and standard library modeling are mature.

Recommendation:

Keep the model internal at first. Generate optional outputs from it, but do not
make `LuaApiManifest` a public runtime API until at least JSON and LuaLS outputs
have stabilized.

## Versioning

Manifest schema versioning should exist from day one and should be independent
from PopLua package versioning.

Recommended fields:

```json
{
  "schema": "poplua.apiManifest",
  "manifestVersion": 1,
  "popluaVersion": "0.x.y"
}
```

Rationale:

- PopLua can ship runtime fixes without changing the manifest schema.
- Manifest schema changes can happen without forcing a PopLua major version.
- Tools can fail fast or choose compatibility code paths based on
  `manifestVersion`.
- JSON, LuaLS, Markdown, and external manifest consumers need a stable contract that is not
  coupled to generated C# implementation details.

Versioning belongs in the canonical model, not only in JSON, because all emitters
and future tools need to know which semantic model they are rendering. Individual
output formats may also have their own renderer version, but that is secondary.

Compatibility rule:

- Additive fields may keep the same manifest version when existing tooling can ignore
  them.
- Semantic changes, field name changes, deleted fields, or changed type meanings
  must increment `manifestVersion`.
- Tooling should reject unknown future major schema versions unless explicitly
  configured to continue.
- Nondeterministic fields such as generation timestamps and machine-local source
  paths are diagnostic metadata and must be omitted from compatibility
  comparisons by default.

## Stable API Identity

Every manifest item that can be linked, documented, filtered, or compared should
have a stable logical id in the canonical model.

Recommended id forms:

```text
module:vec
module:vec.new
module:vec.PI
userdata:vec2
userdata:vec2.length
userdata:vec2.x
userdata:vec2.__add
standard:math.sqrt
diagnostic:PLUA002
```

Stable ids belong in the canonical model, not only emitted JSON. Otherwise each
emitter would invent its own link and comparison identity rules.

Rules:

- Ids are based on Lua-facing names, not C# symbol names.
- Module and userdata ids use their Lua names.
- Member ids use their containing Lua API id plus Lua member name or metamethod.
- Standard library ids use the `standard:` prefix.
- Diagnostic ids use the `diagnostic:` prefix.
- Ids are case-sensitive and must not normalize user-chosen Lua names.
- Id collisions are diagnostics where they would produce ambiguous Lua API.
- C# symbol identity should remain available as secondary source metadata, but
  it is not the stable API identity.

Benefits:

- external compatibility analysis can distinguish "C# refactor with same Lua API" from "Lua API
  breaking rename".
- Markdown cross-links and LuaLS references can share one identity system.
- Capability filtering can preserve stable references.
- Future doc sites can generate deterministic anchors.

Limitations:

- Rename detection still needs an optional compatibility map or user-authored
  alias metadata in the future. The base manifest should not guess that two
  different ids are the same API.

## Canonical Type Model

`LuaApiType` is a foundational model and must not leak raw Roslyn types as the
primary representation. Roslyn type symbols should feed the model, but emitters
must consume canonical Lua-facing types.

Required kinds:

```text
nil
boolean
integer
number
string
userdata
lua-value
variadic
multiple-return
void
unknown
future-extension
```

Recommended fields:

```text
LuaApiType
  Kind
  LuaName
  DotnetName?
  UserdataId?
  ElementType?
  ReturnTypes[]
  IsNullable
  IsLossy
  Notes[]
```

Mapping guidance:

- `bool` -> `boolean`.
- `int`, `uint`, `long`, `ulong` -> `integer`.
- `float`, `double` -> `number`.
- `string` -> `string`.
- `[Userdata]` types -> `userdata` with `UserdataId`.
- `Value` -> `lua-value`, rendered as `any` by LuaLS initially.
- final `Value[]` parameter -> `variadic` with `ElementType = lua-value`.
- `Value[]` return -> `multiple-return` with lossy element information until
  PopLua has richer multiple-return annotations.
- `ValueTask<T>` for async bindings should preserve the Lua-facing type `T` and
  mark the function as async; async-ness is a function property, not a separate
  Lua type.
- `ValueTask` without a result maps to `void` and marks the function as async.
- Multiple-return metadata should use ordered `ReturnTypes[]`. A raw
  `Value[]` return is represented as `Kind = multiple-return`,
  `DotnetName = PopLua.Value[]`, and `IsLossy = true` until typed
  multiple-return annotations exist.
- Future coroutine bridge details such as yield/resume behavior belong on the
  function metadata, not in primitive type mappings.

What does not belong in `LuaApiType`:

- Capability requirements. Capabilities belong on APIs or sandbox profile
  filters, not on primitive/userdata type references.
- Formatter-specific names such as LuaLS `any`; those belong in emitters.
- Runtime marshal helper names.
- Native interop type details.

Userdata references should use stable `UserdataId` plus display `LuaName`. This
lets JSON diffing and Markdown links remain stable even if a C# type is renamed.

## Current Tooling Outputs

The current tooling implementation emits internal C# providers by default for
projects with generated bindings and writes external files from those provider
sources through MSBuild targets.

These providers contain JSON manifest text, LuaLS/LuaCATS-compatible definition
text, and Markdown API documentation text. All three outputs are projections over
the same internal `LuaApiManifest` model. The generator still does not inspect
runtime state, perform runtime reflection, or write arbitrary files directly.

## JSON Manifest

The JSON manifest should be the most complete machine-readable output. It should
be stable, versioned, and independent of generated C# implementation details.

Proposed top-level shape:

```json
{
  "schema": "poplua.apiManifest",
  "manifestVersion": 1,
  "assembly": {
    "name": "MyApp",
    "version": "1.2.3",
    "informationalVersion": "1.2.3+abc123"
  },
  "popluaVersion": "1.0.0",
  "modules": [],
  "userdata": [],
  "libs": [],
  "diagnostics": []
}
```

Module shape:

```json
{
  "id": "module:vec",
  "name": "vec",
  "capability": null,
  "source": {
    "dotnetDisplayName": "MyApp.VecModule",
    "metadataName": "MyApp.VecModule",
    "assemblyName": "MyApp"
  },
  "documentation": {
    "summary": "...",
    "remarks": "..."
  },
  "functions": [],
  "values": []
}
```

Function shape:

```json
{
  "id": "module:vec.new",
  "name": "new",
  "kind": "module-function",
  "async": false,
  "source": {
    "dotnetDisplayName": "MyApp.VecModule.New(double, double)",
    "metadataName": "MyApp.VecModule.New"
  },
  "parameters": [
    { "name": "x", "type": { "lua": "number", "dotnet": "double" } }
  ],
  "returns": [
    { "type": { "lua": "vec2", "kind": "userdata" } }
  ],
  "capability": null,
  "documentation": {
    "summary": "...",
    "parameters": {},
    "returns": "..."
  }
}
```

Userdata shape:

```json
{
  "id": "userdata:vec2",
  "name": "vec2",
  "source": {
    "dotnetDisplayName": "MyApp.Vec2",
    "metadataName": "MyApp.Vec2",
    "assemblyName": "MyApp"
  },
  "setters": false,
  "toString": true,
  "gc": true,
  "methods": [],
  "properties": [],
  "operators": []
}
```

Property shape:

```json
{
  "id": "userdata:vec2.x",
  "name": "x",
  "kind": "property",
  "source": {
    "dotnetDisplayName": "MyApp.Vec2.X",
    "metadataName": "MyApp.Vec2.X"
  },
  "type": { "lua": "number", "dotnet": "double" },
  "readonly": true,
  "writable": false
}
```

Operator shape:

```json
{
  "id": "userdata:vec2.__add",
  "metamethod": "__add",
  "operator": "+",
  "parameters": [
    { "type": { "lua": "vec2", "kind": "userdata" } },
    { "type": { "lua": "vec2", "kind": "userdata" } }
  ],
  "returns": [{ "type": { "lua": "vec2", "kind": "userdata" } }]
}
```

The JSON format should include enough source identity to support future API
diffing but should avoid absolute paths by default. Absolute paths can be an
opt-in diagnostic/debug field.

Schema stability recommendations:

- Prefer nested objects over overloaded strings for identities, types,
  documentation, source symbols, and capabilities.
- Keep `id`, `name`, `kind`, `type`, `capability`, `readonly`, `writable`, and
  `async` semantics stable once version 1 ships.
- Use optional `extensions` objects for experimental or emitter-specific
  metadata.
- Required collections should be present as arrays, even when empty.
- Optional diagnostic details should be absent or `null` according to the schema,
  never encoded as sentinel strings.
- Do not expose generated callback names, stack helper names, native handles, or
  interop implementation details as manifest fields.

## LuaLS / LuaCATS Definitions

PopLua can generate `---@meta` files from the same model.

Feasible mappings:

| PopLua concept | Definition representation |
|---|---|
| Module table | `module_name = {}` plus functions/fields. |
| Module function | `function module_name.func(args) end`. |
| Userdata | `---@class userdata_name`. |
| Userdata method | `function userdata_name:method(args) end`. |
| Userdata property | `---@field readonly x number` where supported. |
| Constants | `---@type number` then `module.PI = nil`. |
| Primitive types | `boolean`, `integer`, `number`, `string`. |
| `Value` | `any` initially. |
| `Value[]` variadic | `...: any` initially. |
| Userdata return | class name. |
| Operators | `---@operator add(userdata_name): userdata_name` etc. |

LuaLS operator support can express many metamethods:

```lua
---@class vec2
---@field readonly x number
---@field readonly y number
---@operator add(vec2): vec2
---@operator sub(vec2): vec2
---@operator unm: vec2
```

Limitations:

- EmmyLua/LuaLS cannot perfectly model sandbox visibility at edit time unless
  separate profile-specific definition files are generated.
- `Value` and multiple return values need conservative `any` or generated
  overload-like annotations.
- Async functions will look synchronous to Lua after the coroutine bridge, but
  metadata should retain `async: true` for documentation/tooling.
- Module capability requirements are not naturally represented in LuaLS; they
  should be comments or profile-specific filtering.
- LuaLS supports readonly fields in annotations, but enforcement is editor-side
  only.

Recommended generated definition strategy:

- Generate one complete definition file for the full API.
- Optionally generate sandbox-profile definition files that omit unavailable APIs.
- Include capability comments:

```lua
--- Requires capability: fs.read
```

## Markdown Documentation

Markdown docs can be generated from the same manifest:

```text
# Modules
## mathx
### mathx.add(left, right)

# Userdata
## vec2
### vec2:length()
### Properties
### Operators
```

XML doc integration should map:

- `<summary>` to short descriptions.
- `<param name="...">` to parameter descriptions.
- `<returns>` to return descriptions.
- `<remarks>` to extended notes.
- `<example>` to examples when present.
- `<exception>` to host-side failure notes when relevant.
- Capability notes to structured capability metadata, not only prose.

Documentation generation should not require runtime execution. It should use
only the generator model.

The canonical documentation model should preserve richer structure than any one
emitter currently needs:

```text
LuaApiDocumentation
  Summary
  Remarks[]
  ParametersByName
  Returns
  Examples[]
  Exceptions[]
  SeeAlso[]
  TypeParameters[]
  CapabilityNotes[]
  RawXml?
  Extensions?
```

Recommendations:

- Preserve structured XML doc sections separately instead of flattening to
  Markdown too early.
- Preserve unknown XML elements in `RawXml` or a future extension bag so docs are
  not destroyed by the first emitter implementation.
- Preserve `<see>`, `<seealso>`, and `<typeparam>` as structured data when
  Roslyn provides it. LuaLS may ignore these initially, but Markdown and future
  documentation sites should not need to recover them from flattened prose.
- Keep capability requirements in API metadata and optionally duplicate
  human-authored explanations in `CapabilityNotes`.
- LuaLS definitions consume summaries, remarks, parameter descriptions, and
  return descriptions today, while Markdown and JSON keep the fuller
  documentation model.

## Sandbox Awareness

Sandbox-aware API visibility has two different sources:

1. PopLua-owned standard Lua libraries.
2. User-generated modules with capabilities.

### Standard Lua Libraries

Trusted sessions currently open standard libraries while untrusted sessions do
not. The generator cannot infer this from user code. PopLua should maintain a
versioned internal standard-library catalog matching the runtime's Lua 5.4 or 5.5
policy.

Proposed model:

```text
Lua54StdLibManifest
  LuaVersion
  PopLuaPolicyVersion
  Globals[]
  Libraries[]
  Profiles[]

LibraryProfile
  Name                  // trusted, untrusted, future custom profile
  IncludedApiIds[]
```

Initial profiles:

- `trusted`: includes the standard libraries opened by `luaL_openlibs`.
- `untrusted`: no standard libraries unless PopLua later exposes safe subsets.

The catalog should be PopLua-owned, not user-authored, so docs and autocomplete
match runtime policy. If future sandboxes allow individual standard libraries,
catalog entries can gain capability tags such as `debug`, `fs.read`, or a
PopLua-specific standard-lib capability.

Ownership model:

- PopLua owns the catalog because standard libraries are runtime policy.
- The catalog should live in the PopLua package, not in user projects.
- User projects may select/filter profiles, but they should not redefine Lua 5.4 or 5.5
  standard APIs as generated module metadata.

Maintenance burden:

- The initial catalog can be deliberately small: globals and libraries PopLua
  actually opens for trusted sessions.
- Changes should be tested against the runtime policy, especially if untrusted
  sessions later gain safe subsets.
- The catalog should have its own policy version because Lua 5.4 or 5.5 and PopLua's
  sandbox policy can evolve independently.

Composition:

- The canonical manifest should include both generated APIs and standard library
  APIs.
- Generated APIs and standard APIs should share `LuaApiType`, stable ids,
  documentation, and capability/profile filtering.
- Standard APIs should be marked with `source = standard-library` so tooling can
  distinguish host APIs from Lua built-ins.
- Profile membership should be represented by stable API ids. Profile-specific
  outputs are projections over the manifest, not copied standard-library API
  objects with independent identities.

### Capability-Based Modules

`[Module(Cap = "...")]` means:

- The API exists in the application.
- The API is registered only when the active sandbox has the capability.
- Documentation should be able to show either the complete API or a filtered
  profile.

JSON should preserve the capability field. Definition and Markdown output should
support:

- Full manifest with capability annotations.
- Filtered manifest for a named sandbox profile.
- Grouping by capability.

Capability ownership rules:

- Capability strings are owned by the host application for generated APIs and by
  PopLua runtime policy for standard-library APIs.
- The manifest records required capabilities; it does not grant, deny, or
  evaluate runtime access.
- Profile filtering is a documentation/tooling projection over manifest ids. It
  must not change `Sandbox` runtime semantics or require named runtime sandboxes.
- Capability changes are compatibility-relevant even when the Lua name and
  function signature are unchanged.

PopLua does not currently have named sandbox profiles, only `Sandbox` instances
and fluent builders. A future manifest system should introduce a build-time
metadata concept rather than trying to serialize arbitrary runtime `Sandbox`
objects:

```xml
<PopLuaApiProfile Include="trusted" Capabilities="*" Libs="trusted" />
<PopLuaApiProfile Include="untrusted" Capabilities="" Libs="untrusted" />
<PopLuaApiProfile Include="plugin" Capabilities="plugin.log;plugin.state" Libs="untrusted" />
```

This keeps runtime sandbox behavior unchanged while allowing tooling to generate
profile-specific outputs.

## Build Integration

The current source generator can only add C# source to the compilation. It should
not directly write arbitrary `.json`, `.lua`, or `.md` files.

There are three realistic output architectures:

### Option A: Generator emits C# metadata, MSBuild task exports files

The generator emits a C# source file containing serialized manifest data or a
strongly typed generated manifest provider. A packaged MSBuild task reads that
metadata from compiler outputs or generated intermediate files and writes JSON,
LuaLS, and Markdown files.

Benefits:

- Keeps all semantic collection in the generator.
- Keeps arbitrary file writes in MSBuild where they belong.
- Preserves package simplicity: one package can include analyzer and targets.
- Avoids runtime reflection.

Risks:

- Requires careful incremental build inputs/outputs.
- Reading generated constants from compiled assemblies would be reflection-like
  and should be avoided. Prefer generated intermediate artifacts or compiler
  item handoff.

### Option B: Incremental generator plus AdditionalFiles plus MSBuild targets

MSBuild passes configuration through analyzer config/options and additional
files. The generator uses those options to decide what C# manifest providers to
emit. MSBuild targets write final artifacts.

Benefits:

- Fits Roslyn incremental generator design.
- Supports build profiles and opt-in output flags.
- Keeps configuration declarative.

Risks:

- AdditionalFiles are inputs, not a natural output channel.
- Still needs an MSBuild target/task for external files.

### Option C: Analyzer plus generator plus build task

Package a generator for semantic collection, analyzers for manifest diagnostics,
and a build task/tool for external outputs.

Benefits:

- Clean separation of diagnostics, generated binding code, and file emitters.
- Most future-proof for external manifest consumers and docs sites.

Risks:

- Highest moving parts.
- More packaging and test surface.
- Easy to overbuild before the JSON schema has proven itself.

Recommended phased integration:

PopLua should converge on Option C, but implement it in Option B-sized phases:

1. Build the internal canonical manifest model inside the generator.
2. Use it for current binding generation.
3. Feed generator options through analyzer config/MSBuild properties.
4. Emit a generated C# manifest provider as a source file when enabled:

   ```csharp
   internal static partial class PopLuaApiManifestProvider
   {
       public const string Json = "...";
   }
   ```

5. Add package MSBuild targets for file outputs after the model is stable:

   ```xml
   <PopLuaApiOutputDir>$(MSBuildProjectDirectory)/artifacts/poplua-api</PopLuaApiOutputDir>
   ```

6. The MSBuild target writes files from generated provider content. It must not
   inspect assemblies through reflection.

Default behavior:

- Binding generation remains default.
- External manifest/definition/doc files remain opt-in.
- Generated providers remain available for manual workflows when the matching
  generation property is enabled.

This avoids unexpected extra files while keeping the single-package experience.

Recommended decision:

- Do not make external files default.
- Do not use runtime reflection to export manifests.
- Do not let each emitter recompute metadata.
- Put build configuration in MSBuild properties and analyzer config options.
- Keep emitted artifacts under `obj/.../poplua-api` by default, with an opt-in
  property for copying to a source-controlled docs/definitions directory.

Implementation contract:

- Semantic collection, validation, and rendering should be separate generator
  phases.
- Binding generation and manifest emission must consume the same validated model.
- Emitter tests should assert model-to-output mapping rather than reparsing
  generated binding C#.
- Build tasks may consume generated intermediate manifest data, but they must
  not load compiled user assemblies to discover APIs.

## Compatibility And Tooling

The manifest can later support:

- external compatibility analysis between two JSON manifests.
- Breaking change detection in CI.
- Documentation sites.
- Editor integrations.
- Capability-specific SDK docs.
- Test assertions that a host exposes exactly the intended script API.

Decisions that help future compatibility:

- Version the JSON schema from the start.
- Preserve stable Lua names and stable type identifiers.
- Include capability and readonly/writable flags.
- Keep source locations optional.
- Do not expose generator-internal C# implementation details as schema fields.
- Record async metadata so tooling can identify coroutine behavior.

Minimum comparison inputs:

- `id` distinguishes removals, additions, and Lua-facing renames.
- `source` distinguishes C# refactors from Lua API changes when `id` is stable.
- `capability` distinguishes sandbox visibility changes.
- `LuaApiType` distinguishes parameter, return, value, and userdata type
  changes.
- `readonly` and `writable` distinguish mutability changes.
- `async` distinguishes coroutine behavior changes.

Breaking-change defaults:

- Removed ids are breaking.
- Parameter count, order, requiredness, or type changes are breaking unless an
  explicit future compatibility annotation says otherwise.
- Return type changes are breaking.
- Writable-to-readonly changes are breaking.
- Added or changed capability requirements are potentially breaking because an
  existing sandbox may no longer see the API.
- Documentation-only changes are non-breaking by default.
- Source-symbol changes with unchanged ids and signatures are non-breaking by
  default.

Current generated provider metadata is sufficient for a first external comparison:

- Stable ids for modules, userdata, functions, values, and operators.
- Canonical `LuaApiType` kind, Lua name, .NET display name, and userdata ids.
- Capability requirements.
- Async/static flags.
- Read-only/writable flags.
- Source symbol identity for diagnostics and review context.
- Structured XML docs for human-readable change reports.
- Opt-in external JSON manifest emission for CI consumption.

Known gaps for external manifest consumers:

- No alias/rename metadata, so renames currently appear as remove/add.
- No standard-library catalog yet.
- No sandbox profile projection metadata beyond raw capabilities.

## Non-Goals

The manifest system must not become:

- Runtime reflection or runtime assembly scanning.
- Runtime Lua API discovery.
- A second binding system.
- A public runtime dependency required to execute scripts.
- A public exposure path for `LuaNative`, `LuaStack`, or other interop internals.
- A sandbox policy mutation mechanism.
- A replacement for generated registration.

Build-time metadata should be derived from the same source-generation model as
bindings. Runtime execution should continue through `Engine`, `Session`,
generated registration code, and `Sandbox`.

## Deferred Ideas

Interpolated-string-handler Lua chunk builders may be a useful future ergonomics
feature, but they should not influence the manifest architecture. They are
runtime authoring helpers, not API surface metadata, and should stay out of the
manifest roadmap until the core manifest, docs, definitions, and diff story are
settled.

## Specification Additions

Recommended v2 specification additions:

- Add an "API Manifest" section after "Generated Code".
- State that PopLua's exposed Lua API is represented by a generator-built
  internal manifest model.
- State that generated bindings, JSON manifests, Lua definitions, and docs must be
  derived from the same model.
- Define JSON schema goals and core fields.
- Define stable id generation rules and source-symbol metadata.
- Define canonical `LuaApiType` mapping rules, including async and
  multiple-return representation.
- Define LuaLS/LuaCATS definition goals and known limitations.
- Define XML documentation mapping.
- Define capability-aware and profile-aware output behavior.
- State that standard Lua library metadata is PopLua-owned and versioned.
- State that external file generation is opt-in.
- State explicit non-goals: no runtime reflection, no runtime API discovery, no
  second binding system, no public interop exposure, and no sandbox policy
  mutation.

## Roadmap Additions

Recommended roadmap sequence:

1. Lock manifest schema versioning, stable id rules, source-symbol metadata, and
   `LuaApiType`.
2. Refactor generator collection into a canonical internal API model.
3. Add XML documentation extraction and preservation tests.
4. Add standard Lua 5.4 or 5.5 library catalog for trusted/untrusted profiles.
5. Add opt-in generated JSON manifest.
6. Add package validation for manifest generation.
7. Add LuaLS/LuaCATS definition generation.
8. Add Markdown documentation generation.
9. Add sandbox profile metadata and filtered outputs.
10. Add documentation site generation.

The async coroutine bridge can proceed independently, but the manifest model
should reserve async metadata now so generated docs and future definitions do not need
a schema break later.
